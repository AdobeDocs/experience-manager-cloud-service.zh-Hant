# 使用通用編輯器建立回應式Forms

使用者可在各式各樣的裝置上存取表單，包括桌上型電腦、平板電腦和智慧型手機。 設計回應式表單可確保為所有使用者提供最佳體驗，不受裝置影響。 本指南說明如何使用通用編輯器針對任何熒幕大小設計、測試和最佳化表單。


回應式表單建立涉及兩個主要活動：

- **回應式測試：**&#x200B;使用裝置模擬器預覽及測試各種熒幕大小的表單。
- **回應式設計：**&#x200B;選擇並實作順暢地適應不同裝置的佈局模式。

**在本指南中，您將學習如何：**

- 在桌上型電腦、平板電腦和行動裝置上測試表單
- 為您的內容選取適當的配置模式
- 套用回應式設計最佳實務
- 疑難排解常見的回應式表單問題
- 針對行動效能最佳化表單

## 為什麼回應式Forms很重要

**使用者體驗影響：**

- 超過60%的使用者在行動裝置上存取表單
- 行動體驗不佳，導致放棄率提高67%
- 回應式表單最多可提高25%的完成率

**企業福利：**

- 較高的表單完成率
- 提高使用者滿意度
- 更優異的協助工具合規性
- 降低開發和維護成本

>[!TIP]
>
> **行動優先方法：**&#x200B;開始為行動裝置進行設計，然後針對較大的熒幕進行增強。 這可確保核心功能在限制最嚴格的環境中運作。

## 第1部分：跨裝置測試Forms

在不同裝置上測試您的表單可協助您在使用者遇到問題之前識別並解決回應式問題。 Universal Editor提供模擬器模式，可模擬各種熒幕大小和方向。

### 如何測試您的Forms

**步驟1：開啟裝置模擬器**

1. 在通用編輯器中開啟您的表單。
2. 按一下工具列中的![模擬器圖示](/help/edge/docs/forms/universal-editor/assets/emulator.png){height=2%，width=2%} **模擬器**&#x200B;圖示。
3. 裝置選取器功能表隨即顯示。

![通用編輯器回應式測試介面](/help/edge/docs/forms/universal-editor/assets/universal-editor-emulator.png)

**步驟2：選取要測試的裝置**

- **案頭** （1200px+寬度）：預設編輯檢視
- **平板電腦** （768px-1199px寬度）： Medium熒幕測試
- **行動裝置** （320px-767px寬度）：小熒幕測試
- **自訂**：指定特定裝置的確切尺寸

**步驟3：測試裝置方向**

使用&#x200B;**熒幕旋轉器**&#x200B;圖示來測試兩者：

- **縱向模式**：標準行動方向
- **橫向模式**：旋轉的平板電腦或手機檢視

### 裝置測試結果

每種裝置型別都會顯示不重複的回應行為：

| **裝置類型** | **熒幕寬度** | **檢查內容** | **常見問題** |
|-----------------|------------------|-------------------|-------------------|
| **案頭** | 1200px+ | 完整版面，所有可見功能 | 過多的空白字元、過寬的版面配置 |
| **平板電腦** | 768px-1199px | 元件棧疊、導覽可用性 | 笨拙的規模、觸控目標問題 |
| **行動** | 320px-767px | 單欄版面配置，縮圖導覽 | 小文字，間距相近的按鈕 |
| **自訂** | 使用者定義 | 裝置特定需求 | 中斷點邊緣案例 |

### 測試工作流程

新Forms的&#x200B;**：**

1. **在案頭檢視中建置：**&#x200B;從完整功能開始。
2. **在平板電腦上測試：**&#x200B;檢查中熒幕的改寫。
3. **在行動裝置上進行驗證：**&#x200B;確保在小型熒幕上的可用性。
4. **修正回應式問題：**&#x200B;視需要調整版面。
5. **重新測試所有裝置：**&#x200B;確認所有大小的修正。

現有Forms的&#x200B;**：**

1. **快速行動檢查：**&#x200B;表單是否適用於手機？
2. **識別問題區域：**&#x200B;備註配置和可用性問題。
3. **系統測試：**&#x200B;徹底測試每個裝置大小。
4. **檔案問題：**&#x200B;追蹤需要修正的專案。
5. **實作修正：**&#x200B;有條不紊地解決問題。

## 第2部分：選取回應式佈局模式

版面配置模式會決定表單內容如何適應不同的熒幕大小。 正確的模式可改善使用者體驗和行動效能。

### 版面配置模式概觀

| **配置型別** | **最適合** | **行動效能** | **複雜性** |
|---------------------|-------------------------------------------|------------------------|----------------|
| **面板配置** | 分類的內容，儀表板樣式表單 | 良好 | 低 |
| **精靈配置** | 多步驟流程、複雜的工作流程 | 極佳 | 中 |
| **收合式選單配置** | 常見問題集樣式內容，可選區段 | 極佳 | 低 |

### 快速決策指南

**使用面板配置時機：**

- 您的內容會分為不同的類別
- 使用者需要同時檢視多個區段
- 內容相對簡單

**使用精靈配置時：**

- 表單具有多個邏輯步驟
- 您想要減少認知負擔
- 行動使用者為主要對象

**使用收合式選單配置時：**

- 此表單包含選填或次要內容
- 空間保育很重要
- 內容可以在邏輯上分組

### 面板版面

**用途：**&#x200B;將相關內容組織成可同時檢視的不同視覺區段。

![面板配置範例](/help/edge/docs/forms/universal-editor/assets/panel-layout.png)

**回應式行為：**

- **桌上型電腦(1200px+)：**&#x200B;個面板並排或格線顯示
- **平板電腦(768px-1199px)：**&#x200B;面板以間距垂直棧疊
- **行動(320px-767px)：**&#x200B;具有清除分割槽符號的單欄配置

**實作步驟：**

1. 使用[面板元件](https://experienceleague.adobe.com/zh-hant/docs/experience-manager-core-components/using/adaptive-forms/adaptive-forms-components/panel)。
2. 在每個面板中將相關欄位分組。
3. 為每個區段新增清除標題。
4. 確保面板之間有足夠的間距。

**最佳實務：**

- 桌上型電腦最多只能使用3到4個面板，以免讓使用者不知所措。
- 對每個面板使用描述性標題。
- 邏輯地將相關欄位分組，以減少認知負載。
- 測試觸控裝置上的面板導覽。

**範例使用案例：**

- **工作應用程式：**&#x200B;個人資訊、教育、經驗、參考
- **產品註冊：**&#x200B;基本詳細資料、技術規格、保固資訊
- **調查Forms：**&#x200B;人口統計、偏好設定、意見回饋、連絡人

### 精靈版面

**用途：**&#x200B;引導使用者逐步完成複雜的程式，減少認知負擔，並提高完成率。

![精靈配置範例](/help/edge/docs/forms/universal-editor/assets/wizard-layout.png)

**回應式行為：**

- **所有裝置：**&#x200B;維持單一步驟焦點，以獲得最佳的行動體驗。
- **步驟內容：**&#x200B;可在每個步驟中調整（棧疊或並排）。
- **導覽：**&#x200B;具有足夠間距的觸控式按鈕。
- **進度指示器：**&#x200B;已針對熒幕大小適當縮放。

**實作步驟：**

1. 使用[精靈元件](https://experienceleague.adobe.com/zh-hant/docs/experience-manager-core-components/using/adaptive-forms/adaptive-forms-components/wizard)
2. 將複雜的表單分成邏輯步驟（3-7個步驟是最佳選擇）。
3. 包含使用者方向的進度指示器。
4. 提供清除的導覽控制項（下一步、上一步、儲存）。

**行動裝置最佳化：**

- 導覽按鈕請使用大型觸控目標（最少44畫素）。
- 確保小熒幕上的步驟指示器清晰可見。
- 限制每個步驟的欄位數以減少捲動。
- 啟用自動儲存以防止資料遺失。

**最佳實務：**

- 確保邏輯步驟進度 — 每個步驟都應建立在上一個步驟上。
- 使用清楚的步驟標題，讓使用者知道將會發生什麼事。
- 在每個步驟驗證輸入，以及早擷取錯誤。
- 允許使用者向後導覽，以檢閱或編輯資訊。

**範例使用案例：**

- **保險索賠：**&#x200B;個事件→證據→個人→檢閱
- **帳戶設定：**&#x200B;基本資訊→喜好設定→安全性→確認
- **訂單處理：**&#x200B;產品→送貨→付款→摘要

### 摺疊式版面

**用途：**&#x200B;將內容組織成可摺疊的區段，以節省空間，非常適合選用或次要資訊。

![收合式選單配置範例](/help/edge/docs/forms/universal-editor/assets/accordion-layout.png)

**回應式行為：**

- **優異的行動效能：**&#x200B;只顯示相關內容。
- **觸控最佳化的標頭：**&#x200B;容易點選和展開區段。
- **平滑動畫：**&#x200B;提供互動的視覺回饋。
- **有效空間：**&#x200B;最小化所有裝置上的捲動。

**實作步驟：**

1. 使用[摺疊式元件](https://experienceleague.adobe.com/zh-hant/docs/experience-manager-core-components/using/adaptive-forms/adaptive-forms-components/accordion)。
2. 在每個區段中群組相關的選用內容。
3. 使用描述性區段標題。
4. 設定適當的預設開啟/關閉狀態。

**行動裝置優勢：**

- 摺疊未使用的區段以減少捲動。
- 具備觸控功能的互動功能，可自然展開/收合手勢。
- 載入更快 — 只顯示作用中的內容。
- 改善焦點 — 使用者只看到他們需要的內容。

**最佳實務：**

- 使用清除區段標題，讓使用者在展開之前先瞭解內容內容。
- 依邏輯將每個區段中的相關內容分組。
- 視需要設定重要區段以開始展開。
- 提供簡短章節預覽，協助使用者決定要展開的內容。

**範例使用案例：**

- **產品組態：**&#x200B;基本→進階→附件→支援
- **常見問題集Forms：**&#x200B;帳戶→帳單→技術資→一般
- **設定Forms：**&#x200B;隱私→通知→外觀→進階

## 第3部分：回應式設計最佳實務

### 依裝置型別的最佳實務

+++行動最佳化(320px-767px)

**基本作法：**

- 對所有內容使用單欄式配置。
- 提供大型觸控式按鈕（最低為44px高度）。
- 使用清楚的「上一頁/下一頁」選項簡化導覽。
- 將每個區段內的捲動最小化。
- 在第一個欄位上自動對焦以顯示鍵盤。

**欄位特定准則：**

- **文字輸入：**&#x200B;全寬，具有足夠的內距。
- **下拉式清單：**&#x200B;使用原生選取元素以獲得更佳的觸控體驗。
- **日期選擇器：**&#x200B;使用原始日期輸入來與行動裝置相容。
- **檔案上傳：**&#x200B;提供大型且清除的上傳區域。

+++

+++平板電腦最佳化(768px-1199px)

**配置策略：**

- 相關欄位使用雙欄版面配置。
- 測試直向和橫向方向。
- 支援觸控和滑鼠互動。
- 提供更大的內容區域，同時保持可讀性。

+++

+++桌上型電腦最佳化(1200px+)

**進階功能：**

- 使用多欄配置以有效利用空間。
- 為進階使用者提供鍵盤快速鍵。
- 實作互動式意見反應的暫留狀態。
- 提供包含詳細錯誤訊息的進階驗證。

+++

## 完整的疑難排解

### 配置問題

+++行動裝置上的表單版面分隔符號

**常見原因：**

- 不縮放的固定寬度元素
- 專為案頭優先版面配置設計的CSS
- 使容器溢位的影像或內容

**解決方案：**

- 確定影像和容器可縮放至熒幕大小。
- 使用行動優先設計，並具備漸進式增強功能。
- 使用裝置模擬器和實際裝置進行測試。
- 使用彈性大小調整，而非固定維度。

+++

+++觸控目標太小

**常見原因：**

- 小於44px × 44px的按鈕
- 互動式元素放置得太接近
- 自訂CSS覆寫觸控易記預設值

**解決方案：**

- 請確定所有互動式元素都至少為44px×44px。
- 新增按鈕和連結之間的間距。
- 測試觸控式互動功能是否與實際的手指相容，而不只是滑鼠。
- 增加觸控目標區域，讓點選更輕鬆。

+++

+++內容溢位問題

**常見原因：**

- 無法繞排的長文字或標籤
- 固定寬度容器
- 未正確縮放的影像

**解決方案：**

- 啟用長內容的文字繞排。
- 使用可適當縮放的回應式影像。
- 實作適應內容的彈性版面配置。
- 測試各種內容長度。

+++

### 效能問題

+++在行動裝置上載入緩慢

**常見原因：**

- 大型影像未針對行動裝置最佳化
- JavaScript過度執行
- 太多表單欄位同時載入

**解決方案：**

- 針對不同的熒幕大小最佳化影像。
- 僅在需要時載入非關鍵內容。
- 使用技巧加速行動載入。
- 將第三方指令碼和Widget最小化。

+++

### 測試和驗證問題

+++模擬器與真實裝置的差異

**常見原因：**

- 瀏覽器特有的轉譯差異
- 觸控與滑鼠互動的差異
- 網路速度變化

**解決方案：**

- 儘可能在實際裝置上進行測試。
- 使用多種瀏覽器進行模擬器測試。
- 在測試期間模擬不同的網路速度。
- 與目標環境中的真實使用者進行驗證。

+++
